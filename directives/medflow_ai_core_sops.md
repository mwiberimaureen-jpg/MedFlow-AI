## 1. Ward Round Note Format

**This has regressed twice. Do not let it regress again.**

The ward round note is generated by `generateWardRoundNote` in `lib/openrouter/client.ts`. The prompt lives in `WARD_ROUND_NOTE_PROMPT`.

### Section order
Demographics line → Chief Complaint → Past Medical History (if relevant) → Family History (if relevant) → Examination at Admission → Management at Admission → PLAN

### Chief Complaint — the most common failure point
- **SYNTHESIZE, do NOT transcribe**
- Include: onset, duration, frequency, severity, and the key features that drive the impression (e.g. convulsion count, convulsion duration, tracheal deviation, LOC, photophobia, fecal/urinary incontinence)
- Omit: narrative filler, redundant negatives, irrelevant background
- All clinically significant numbers must be retained: vitals, doses, volumes, episode counts
- The full note must be readable in under 2 minutes

**WRONG:** Copying the HPI text verbatim into Chief Complaint  
**CORRECT:** "3 episodes of generalised tonic-clonic seizures over 6 hours, each lasting ~2 min; post-ictal confusion; no fever, no neck stiffness"

### Other rules
- Include ALL investigations sent, with results if documented
- Include ALL procedures (IV access, NGT, catheter, nasal packing, etc.)
- Include ALL drugs with dose/route/frequency
- Write in brief clinical phrases — no full sentences, no "the patient"
- **NEVER add impressions, diagnoses, or differentials** — those come from the AI Assessment section
- If format regresses, fix `WARD_ROUND_NOTE_PROMPT` — do not wait for the user to report it

## 2. Dark / Light Mode — Persistence Rules

Dark mode preference is persisted in **two places**:
1. **Supabase** — `users.dark_mode` (boolean column)
2. **localStorage** — key `theme`, value `"dark"` or `"light"`

### On login / page load
`DashboardShell` fetches `dark_mode` from Supabase. If the value is a boolean, it:
- Applies/removes the `dark` class on `document.documentElement`
- Writes to localStorage as fallback

### On toggle
`handleThemeToggle` in `DashboardShell.tsx` calls `/api/profile` PATCH with `{ dark_mode: isDark }` and updates localStorage immediately.

### Do NOT
- Use only localStorage (preference is lost on mobile login from a new device)
- Use only Supabase (introduces latency on first render)
- Clear or reset the preference on navigation or re-render
- Default everyone to light mode — always hydrate from stored preference

## 3. Profile Picture / Avatar — Never Clear It

Avatar is stored in Supabase Storage bucket `avatars` (public) and saved to `users.avatar_url` (TEXT column).

### Invariants
- The `avatar_url` column must exist — migration: `supabase/avatar_migration.sql`
- Upload path: `{user_id}/avatar.jpg` (always JPEG, compressed to max 400×400px client-side)
- Append `?t={timestamp}` cache-buster to the URL after every new upload
- `app/dashboard/layout.tsx` fetches the user row (incl. `avatar_url`) **with the admin/service-role client** and passes `avatarUrl` down to `DashboardShell` as a prop — DO NOT switch this to a client-side `supabase.from('users').select(...)` query. A fresh/refreshing session JWT makes RLS (`auth.uid() = id`) silently return zero rows, so the avatar comes back null and the placeholder shows even though the DB has the value (this is the exact "stale JWT → RLS returns null" trap the layout already documents and routes around for `terms_version` — same fix, same reason).
- After a new upload, the Settings page dispatches `window.dispatchEvent(new CustomEvent('medflow:avatar-updated', { detail: { avatarUrl } }))` (same `medflow:` event-prefix convention as `medflow:streak-updated`) — `DashboardShell` listens and updates immediately, carrying the URL directly so no DB round-trip (and no RLS trap) is involved. This is required because the dashboard layout persists across client-side navigation and won't otherwise refetch.
- `/api/profile` PATCH handles `avatar_url` updates — errors must be surfaced, never silently swallowed

### Do NOT
- Clear `avatar_url` on navigation, re-render, or session refresh
- Overwrite `avatar_url` unless the user explicitly uploads a new photo
- Show a default/blank avatar when `avatar_url` exists in DB
- Fetch `avatar_url` client-side via the user's own JWT/RLS path — always seed it server-side with the admin client (see Invariants above); this was the actual root cause of the "photo disappears" bug on 2026-06-08

## 4. Clinical Summary — Faithful Transcript of the History

The AI clinical summary reports **everything the user documented in the history, and nothing else**.

The single rule: **if the user wrote it, include it. If the user did not write it, omit it.**

This means:
- If the user included test results and values → include them in the summary
- If the user stated a diagnosis or impression → include it as written
- If the user did not mention a test result → do not add it
- If the user did not state a diagnosis → do not generate one

### Required content (in order)
1. Name, age, sex, parity (if OB/GYN), Day N of admission
2. Chief complaint and relevant background — exactly as documented
3. Examination findings as documented (appearance, vitals, key exam findings)
4. Investigations mentioned in the history — include results and values if the user documented them
5. Any diagnoses or impressions the user stated — include them as written
6. Treatments/procedures already given — every drug with dose/route/frequency, every procedure
7. Any pending plans explicitly stated in the history

### Absolute prohibitions
- Never add anything not written in the history
- Never generate AI diagnoses, inferred results, or interpretations beyond what the user stated
- Never omit something the user explicitly wrote

### Length
One paragraph for most cases. Two only if the case is genuinely complex.

## 5. Clinical Reasoning Rules

### SIRS vs Sepsis
- Do NOT diagnose "sepsis" without documented end-organ damage (SOFA criteria)
- Use "SIRS secondary to [source]" if there is no organ damage
- Forbidden phrases (even if qualified): "evolving sepsis", "septic picture", "sepsis-like picture"
- The `sanitizeAnalysis()` function catches most slip-throughs, but check the prompt in `lib/openrouter/client.ts` if this regresses

### Anemia thresholds (AMBOSS/WHO)
| Population | Anemia threshold | Mild | Moderate | Severe |
|---|---|---|---|---|
| Non-pregnant women | HB < 12.0 | 11–11.9 | 8–10.9 | < 8 |
| Men | HB < 13.0 | 11–12.9 | 8–10.9 | < 8 |
| Pregnant | HB < 11.0 | 10–10.9 | 7–9.9 | < 7 |

- HB ≥ 12.0 in a non-pregnant woman = NORMAL — never call it anemia
- Never call HB ≥ 8 "severe anemia" in any population
- Source: AMBOSS only. No UpToDate, Medscape, BMJ, or WHO references

## 6. Admission Workflow — Page Layout

Fixed order — do not reorder:

1. **Clinical Summary** (top)
2. **Past Day Summaries** (collapsible, collapsed by default)
3. **Current Day N of Admission** (active card)
4. **Discharge**

### Day N card — internal section order
1. **Assessment** (collapsible) — sub-sections: Follow-up Questions, ROS, Vitals, Physical Exam, Impression, Test Interpretation, Differential Diagnoses, Confirmatory Tests, Management Plan, Complications. Each sub-section has its own Submit + Edit button.
2. **Checklist** (collapsible) — brief checkboxes only, no badges or descriptions. Auto-checks from Assessment input.
3. **Day Notes Summary** (collapsible) — editable textarea formatted as clinical documentation. Final submit button lives here. Does NOT auto-submit.

### Day Notes Summary format
HPI as narrative (no header) → "Review of Systems:" → "Vital Signs:" → "Physical Examination:" → "Investigations:" → "PLAN:" (all caps, each on a new line)

### Always-visible clinical reasoning sections
Impression, Test Interpretation, Differential Diagnoses, and Complications must always render — even without AI content. Show fallback text if content is not available.

### Day number calculation
```
Day = Math.floor((today - admissionDate) / 86400000) + 1
```
Day 1 = the admission date itself.

## 7. Always Generate Analysis on Admission

When a patient history is submitted or an admission is analysed:
- The clinical summary must be generated automatically — do not require a manual trigger
- The ward round note must be generated and cached on `user_feedback` at the same time
- Round notes must also be bulk-regenerable via the admin route

If analysis generation is gated behind a button that shouldn't require clicking, that is a regression — fix it.

## 8. UI State Persistence — localStorage Rule

Any user input or UI state that should survive page navigation **must** be persisted to localStorage keyed by patient ID. React state alone resets on navigation.

This applies to: draft form content, submitted/edit status, checklist state, section open/closed state.

Pattern: on every state change → write to `localStorage.setItem(\`key-{patientId}\`, JSON.stringify(value))`. On mount → hydrate from localStorage before rendering.

## 9. Learning Sparks — Depth Standard

Target level: **final-year medical student / junior doctor (intern-level is too low)**.

Each spark must cover one of:
- A gray area or clinical decision that experienced clinicians get wrong
- A contraindication trap or drug interaction
- Decision-hierarchy reasoning (when to choose A over B)
- A non-obvious clinical rule with a mnemonic if one exists

Do NOT produce:
- Basic textbook recall ("What are the 4 criteria for X?")
- Introductory definitions
- Anything a second-year student would already know

Model: `anthropic/claude-3.5-haiku` (fast, cheap). Prompts are in the learning sparks generation file.

## 10. AI Output Formatting Rules (for Notion to-do lists)

These rules apply to any content written to Notion pages (patient to-do lists, analysis outputs):

**Allowed:**
- Double underscores for bold: `__bold text__`
- Numbered lists: `1.`, `2.`, `3.`
- Dash bullet points: `- item`
- CAPITALIZED WORDS for headers

**Forbidden:**
- Asterisks `*` anywhere
- Hashtags `#` anywhere
- Markdown bold `**text**`
- Markdown headers `##`

Verify before submitting: zero asterisks, zero hashtags.

## 12. Lab Test Interpretation — Do Not Request Completed Tests

When a lab value is mentioned (e.g., "HB is 8.5"), the entire panel is considered done. Do not ask for "complete CBC", "LFTs", "U&Es", etc.

Only abnormal values are reported. Any unmentioned parameter = normal.

Only request tests that provide genuinely new information not already available. For monitoring, name the specific parameter — not the panel.

Full rules: `directives/patient_history_analysis.md` — Lab & Test Interpretation section.

## 13. Patient Rotation Assignment

Do NOT assign all patients to the same rotation by default.

Determine each patient's rotation from their working diagnosis and admission reason:
- Internal Medicine, OB/GYN, Surgery, or Pediatrics

Store rotation in `patient_histories.metadata` JSONB column. Always use `{ ...(patient.metadata || {}), rotation: value }` spread pattern to avoid overwriting other metadata fields.

## 14. OpenRouter Model Names

Use the full namespaced model ID:
- `anthropic/claude-sonnet-4` — main analysis model
- `anthropic/claude-3.5-haiku` — fast tasks (QA checks, learning sparks)

Do NOT use bare model names like `claude-sonnet-4` — they will fail.

## 15. Supabase JSONB Safety

When updating JSONB fields, always use the spread pattern:
```js
{ ...(patient.metadata || {}), rotation: value }
```
Bare `jsonb_set` on a NULL column returns NULL. Always guard with `COALESCE(metadata, '{}')` in raw SQL.

### Review email forwarding to medflowai.ke@gmail.com
`app/api/review/route.ts` saves every review to the `reviews` table AND, if `RESEND_API_KEY` is set, emails it to `medflowai.ke@gmail.com` via Resend (`onboarding@resend.dev` sender — no custom domain needed to receive).

**Critical gotcha (cost ~30 min to debug on 2026-06-14):** Vercel environment variable names are case-sensitive. The code reads `process.env.RESEND_API_KEY` (uppercase). If the Vercel env var is saved as `resend_api_key` (or any other casing), `process.env.RESEND_API_KEY` is `undefined`, the `if (resendKey)` check is falsy, the Resend call is silently skipped, and `/api/review` still returns `{"success": true}` — there is no error anywhere. If reviews aren't arriving by email, check the exact casing of the env var name in Vercel Settings → Environment Variables first, before anything else.

After adding/editing `RESEND_API_KEY` in Vercel, a new Production deployment must run (redeploy) before it takes effect.

## 17. Learning Sparks — Condition Pool, Repeat Policy & Streak Sync

Generation lives in `app/api/learning-spark/today/route.ts` (pool building) and `generateLearningSpark` in `lib/openrouter/client.ts` (prompting).

### Condition pool — ALL patients, not just recent ones
- `recentAnalysesRaw` fetches up to **500** analyses (was 60), `allHistories` fetches up to **200** patient histories (was 30) — these caps exist only as a sanity ceiling, not a "recent N patients" restriction.
- `latestPerPatient` keeps only the most-recent analysis per `patient_history_id` so a long admission (many days of analyses for one patient) can't dominate the pool and crowd out other patients.
- If the "always the same condition" complaint returns, check whether one of these limits is being hit before the dedup — i.e. whether the patient pool is large enough that 500 analyses no longer covers everyone.

### Source categories mined per analysis
`extractConditions()` pulls from, per analysis:
- `analysis.summary` (the faithful clinical-summary text) — both `CONDITION_PATTERN` matches (diagnoses/conditions/labs as documented) and `EXAM_FINDINGS_PATTERN` matches (e.g. Murphy's sign, tracheal deviation, rebound tenderness, raised JVP — tagged `"(exam finding)"` for spot-diagnosis teaching moments)
- `## Impression(s)`, `## Differential Diagnoses`, `## Possible Complications`, `## Test Interpretation`, `## Confirmatory Tests`, and drug names from the Recommended Plan — all from `raw_analysis_text` as before

`CONDITION_PATTERN`, `LAB_PATTERN`, and `EXAM_FINDINGS_PATTERN` are module-level regexes shared between `extractConditions` (analyses) and `extractConditionsFromHistories` (patient histories) — add new terms to these constants, not inline copies.

### Repeat policy — weekly, not a flat 14-day ban
- `daily_learning_sparks` is queried for the last **60 days** and split by `getWeekStart()` (Monday-start) into:
  - `thisWeekTopics` — **hard avoid**. The same underlying condition (any angle: renamed, restaged, different drug/complication/test) is off-limits until next week.
  - `earlierTopics` — **soft avoid**. May repeat on a different week, but only as a last resort and from a genuinely new angle.
- `generateLearningSpark(format, conditions, config, thisWeekTopics, earlierTopics)` — if this signature changes, update both call sites (route + any future callers) and keep the hard/soft distinction in the prompt; collapsing back to one flat list re-introduces the "always HIE" bug.

### Streak "forgetting the day" — race condition fix
`DailyLearningSpark.tsx`'s mount effect fetches `/api/learning-spark/streak` and merges it with local state. **Do not merge against the `local` variable captured at mount** — `handleInteraction` can update localStorage (and React state) while that fetch is still in flight, and merging against the stale mount-time snapshot reverts today's streak increment, which looks like the app "forgot" what day the streak is on. The fix: re-read via `loadState()` (`fresh`) at the moment the fetch resolves, and merge against `fresh`, not `local`.

### `format_type` constraint — RESOLVED 2026-06-14
The `daily_learning_sparks_format_type_check` constraint had never been migrated from the old `('quiz','mystery','myth','flashcards')` values to the current `('senior_asks','quick_teach','know_your_drugs','clinical_twist')`. Result: **every spark insert silently failed for ~3 months** (only one stale row existed, from 2026-03-18, format_type='mystery') — every spark got `id: 'temp'`, never cached, regenerated via the AI on every page load, and the star feature broke (see below).

Fixed by running `supabase/spark_format_migration.sql` in the Supabase SQL Editor on 2026-06-14 (delete stale rows BEFORE adding the new constraint — `ADD CONSTRAINT` validates existing rows immediately, so adding it first throws `23514`). Confirmed: table is now empty and accepts the new format_type values. **Do not re-run this migration** — if sparks stop caching again, the constraint is not the cause; look elsewhere first.

### Star button permanently disabled by stale 'temp' id — fixed 2026-06-14
Because every failed insert returned the same `id: 'temp'`, the first time a user starred a temp spark, `'temp'` got written into `state.starredSparks` in localStorage. From then on, `isStarred = starredSparks.includes(spark.id)` was true for EVERY future spark (also `'temp'`), permanently disabling the Save button.

Fix in `DailyLearningSpark.tsx`:
- On mount, strip any `'temp'` entry from persisted `starredSparks` (self-heal for users who hit this before the format_type fix)
- `isStarred` for `spark.id === 'temp'` is tracked via a session-only `tempStarred` flag, never via persisted `starredSparks` — reset whenever a new spark loads
- With the format_type constraint now fixed, `spark.id` should be a real UUID going forward and this code path shouldn't trigger — but the guard stays as defense-in-depth in case an insert ever fails for an unrelated reason

## 18. Data Retention — Basic Plan Auto-Delete & Trash Purge

Migration: `supabase/basic_plan_retention_migration.sql` (run once in the Supabase SQL Editor — adds two pg_cron jobs).

### Rule
- **Basic plan users** (`subscriptions.plan_type = 'basic'`, `status = 'active'`): any patient history that has **never been edited** since creation (`updated_at = created_at`) and is **not starred** (`is_starred = false`) is auto-moved to trash (`deleted_at = now()`) once it is **older than 1 month** (`created_at < now() - 30 days`).
  - Starring (`StarPatientButton.tsx` → `POST /api/patients/[id]/star`) exempts a history from this regardless of age or edit status.
  - "Edited" = the user went through the edit flow (`PATCH` with `action: 'edit'` or `patient_name`/`history_text` fields) at least once — that's the only code path that sets `updated_at` on `patient_histories`. Rotation changes, status transitions (analyzing/completed/error), and regeneration do NOT bump `updated_at`, so they don't count as "edited".
  - Pro plan and free-trial users are unaffected by this job (existing 90-day `cleanup_old_patient_histories` job from `auto_delete_migration.sql` still applies to everyone who hasn't opted out).
- **Trash retention is 3 days** (was 7 — `components/patients/TrashSection.tsx` updated to match). Anything with `deleted_at` older than 3 days is permanently (hard) deleted, cascading `todo_items` → `analyses` → `patient_histories`.

### Cron schedule (staggered after existing nightly jobs)
- `cleanup-old-patient-histories` — 03:00 UTC (90-day, all users, existing)
- `cleanup-unedited-basic-plan-histories` — 03:10 UTC (1-month, basic plan, un-edited+unstarred — new)
- `cleanup-expired-trial-data` — 03:30 UTC (30-day trial purge, existing)
- `purge-trashed-patient-histories` — 03:40 UTC (3-day trash hard-delete — new)

Both new functions log to `audit_logs` (`patient.auto_delete_basic_plan`, `patient.trash_purge`) with row counts for auditing.

### Status — RESOLVED 2026-06-15
`supabase/basic_plan_retention_migration.sql` was run successfully in the Supabase SQL Editor on 2026-06-15. Both cron jobs are registered and will run nightly going forward. **Do not re-run this migration** — `CREATE OR REPLACE FUNCTION` + the unschedule/schedule blocks are idempotent if it ever needs to be re-applied (e.g. after editing the function bodies), but a fresh run isn't needed for normal operation.

To apply the policy to **existing accounts immediately** (instead of waiting for the next 03:10/03:40 UTC run), run once in the SQL Editor:
```sql
SELECT public.cleanup_unedited_basic_plan_histories();
SELECT public.purge_trashed_patient_histories();
```
Each returns the number of rows affected (soft-deleted / hard-deleted respectively).
